# Delta Patch Binary Format

This document describes the binary delta patch format used by the Redball update system for efficient incremental updates.

## Overview

Delta patches allow clients to update from one version to another by downloading only the differences between versions, rather than the entire new version. This significantly reduces bandwidth usage and update times.

## Format Specification

### File Structure

```text
┌─────────────────────────────────────┐
│ Header (64 bytes)                   │
├─────────────────────────────────────┤
│ Metadata Section (variable)         │
├─────────────────────────────────────┤
│ Compressed Delta Data (variable)    │
└─────────────────────────────────────┘
```

### Header (64 bytes)

| Offset | Size | Field       | Description                           |
| ------ | ---- | ----------- | ------------------------------------- |
| 0      | 4    | Magic       | `0x52425054` ("RBPT" - Redball Patch) |
| 4      | 2    | Version     | Format version (current: 1)           |
| 6      | 2    | Flags       | Bit flags (see below)                 |
| 8      | 8    | OldFileSize | Size of source file in bytes          |
| 16     | 8    | NewFileSize | Size of target file in bytes          |
| 24     | 8    | PatchSize   | Size of compressed delta data         |
| 32     | 32   | OldFileHash | SHA-256 hash of source file (hex)     |

### Flags (2 bytes)

| Bit  | Description                                                 |
| ---- | ----------------------------------------------------------- |
| 0    | Compression enabled (1 = gzip compressed, 0 = uncompressed) |
| 1    | Checksum verification required                              |
| 2-15 | Reserved for future use                                     |

### Metadata Section

JSON-encoded metadata following the header:

```json
{
  "oldFileHash": "sha256-hex-string",
  "newFileHash": "sha256-hex-string",
  "oldFileSize": 1234567,
  "newFileSize": 1234890,
  "patchSize": 45678,
  "compressionRatio": 0.037,
  "createdAt": "2026-04-07T06:10:00Z",
  "algorithm": "bsdiff"
}
```

### Delta Data Section

The delta data is compressed using gzip and contains the binary diff generated by the bsdiff algorithm.

## Patch Generation

Server-side patch generation (Node.js):

```javascript
const { generatePatches } = require('./lib/delta-patches');

// Generate patch between two versions
const patches = await generatePatches(
  '/path/to/old/version',
  '/path/to/new/version',
  '/path/to/output/patches'
);
```

## Patch Application

Client-side patch application (C#):

```csharp
using Redball.Core.Updates;

// Verify old file hash before applying
var oldFileHash = HashUtility.ComputeFileHash(oldFilePath);
if (oldFileHash != patch.OldFileHash)
{
    throw new InvalidOperationException("Source file hash mismatch");
}

// Apply patch
var patchData = await File.ReadAllBytesAsync(patchFilePath);
var newData = await DeltaUpdateService.ApplyPatchAsync(oldData, patchData);

// Verify new file hash after applying
var newFileHash = HashUtility.ComputeBytesHash(newData);
if (newFileHash != patch.NewFileHash)
{
    throw new InvalidOperationException("Patched file hash mismatch");
}

await File.WriteAllBytesAsync(newFilePath, newData);
```

## Security Considerations

1. **Hash Verification**: Always verify both source and target file hashes
2. **Size Limits**: Reject patches that would produce files larger than expected
3. **Path Sanitization**: Validate all file paths to prevent directory traversal
4. **Signature Verification**: Patches should be signed and verified before application

## Performance Characteristics

- **Compression Ratio**: Typically 3-10% of full file size for binary updates
- **Memory Usage**: Requires ~2x the size of the larger file in memory during application
- **CPU Usage**: Moderate - decompression and diff application are CPU-bound

## Error Handling

Common error scenarios:

| Error Code           | Description                     | Recovery Action                   |
| -------------------- | ------------------------------- | --------------------------------- |
| `HASH_MISMATCH_OLD`  | Source file hash doesn't match  | Download full version             |
| `HASH_MISMATCH_NEW`  | Patched file hash doesn't match | Re-download patch or full version |
| `CORRUPT_PATCH`      | Patch file is corrupted         | Re-download patch                 |
| `INSUFFICIENT_SPACE` | Not enough disk space           | Free space and retry              |

## Compatibility

- **Minimum Client Version**: Redball 2.0.0
- **Supported Platforms**: Windows (WPF), Linux (GTK4)
- **Maximum Patch Size**: 500 MB (configurable)
- **Maximum File Size**: 2 GB

## Example Workflow

1. **Server**: Generate patch from v2.1.0 to v2.2.0

```bash
   node update-server/lib/delta-patches.js generate \
     --old releases/2.1.0/Redball.exe \
     --new releases/2.2.0/Redball.exe \
     --output releases/2.2.0/patches/2.1.0-to-2.2.0.patch
   ```

1. **Client**: Check for updates

   ```http
   GET /api/update/windows/stable?currentVersion=2.1.0
   ```

2. **Server**: Respond with patch availability

   ```json
   {
     "version": "2.2.0",
     "patchAvailable": true,
     "patchUrl": "/downloads/2.2.0/patches/2.1.0-to-2.2.0.patch",
     "patchSize": 4567890,
     "fullUrl": "/downloads/2.2.0/Redball.exe",
     "fullSize": 123456789
   }
   ```

3. **Client**: Download and apply patch
   - Download patch file
   - Verify old file hash
   - Apply patch
   - Verify new file hash
   - Replace old file with new file
